# /callbell-plan

The planning session: an idea goes in, a project and its work packages come out. Then this is done and the
work grows in conversation from there.

Planning is a **conversation, not a form**. Never hand the user a template to fill in. The checklist below
is yours: test what they gave you against it, and ask about what is missing.

## Ask, this whole phase

Questions are the point of this phase, not a failure of it. Do not guess your way to a plan that reads
well and means nothing.

- **State your assumptions** rather than burying them. If a requirement has two readings, put both up and
  let the user pick; do not silently choose one.
- **Push back** on what is unclear or contradictory. A plan built on a misreading costs more than the
  question would have.
- **Only cosmetic calls are yours** during planning (wording, naming, ordering that changes nothing).

## 1. Establish what kind of idea this is

Do this first, because it decides how you work:

- **Mature in the user's head:** your job is to **extract**, not to co-invent. Ask, mirror it back, and
  keep your own ideas out unless asked. Adding here drifts the result away from what they actually want.
- **Half-formed:** genuinely think along. Offer options, name trade-offs, bring a recommendation.
- **Generated by another agent** (a plan, a spec, an analysis handed to you): be most skeptical. It is
  checked plausibility, not checked intent, and it reads more finished than it is. Walk it through with the
  user rather than adopting it.

## 2. Clear these axes before you cut

Your checklist, not the user's. Ask about what their input does not already cover:

- **Outcome:** what exists afterwards that does not exist now?
- **Why:** what is it for? Without this, no package can decide anything on its own.
- **Out of scope:** what is explicitly not part of this? The boundary is the useful half.
- **What is already there:** what gets reused rather than rebuilt? Check the repo, do not assume greenfield.
- **Verification:** how is each result checked? Something observable, not "it looks right".
- **Risk:** what is unclear, what could invalidate the plan?

## 3. Cut into work packages

A **package is a task file** (`task-<slug>.md`, template in `__callbell__/templates/`), and it carries why,
scope, approach, and definition of done: enough that it can be worked without further questions. This is
not a checkpoint list ("step 3 of 7"); each package says "here is everything you need".

- **Size it to one session**, before context runs out. That is the span in which an agent plans and works
  coherently. **If you cannot estimate the size, the work is not understood: cut it.**
- **A package never names another package.** Order and dependency go in the roster only (`BACKLOG.md`, or
  the project `index.md`). A later package does not read an earlier one: it needs its *result*, and that
  lives in the repo. Writing a file name into a task body *is* an instruction to read it, so do not write
  it.
- **Several related packages** are a reason to suggest a project folder `__callbell__/backlog/<project>/`.
  Suggest it, explain it in a sentence, and create it **only after approval** (see `callbell-governance`).
  Inside a large project, group in the `index.md` with headings (Backend, Frontend). A heading is grouping
  enough; it never becomes a file level.
- New packages start `status: draft`. **Draft is where assumptions die:** shape them with the user until
  they hold, then they go `active`.

## Done

When the packages stand and no questions are left, this session is over: say so, and stop planning. From
here the work grows in conversation, and re-cutting an active package is normal (say when you do it, never
silently).

Note that a skill cannot switch itself off: this text stays in context until it is compacted away. The
exit is about behavior, not about freeing anything up.
